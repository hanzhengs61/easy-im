### 通信协议：HTTP REST + gRPC + WebSocket

```
HTTP REST
--  适用场景：客户端 ↔ API网关
--  优点：简单通用，浏览器原生支持   
--  缺点：头部冗余，无状态           
--  用途：用户登录、文件上传等       
                               
gRPC
--  用场景：服务间内部调用          
--  优点：高性能、强类型、流式传输
--  缺点：浏览器不支持，调试复杂     
--  用途：User服务→Message服务      

WebSocket                          
--  适用场景：实时双向通信           
--  优点：低延迟、全双工、持久连接  
--  缺点：无状态，需自行管理连接    
--  用途：聊天消息推送、在线状态  
 
协议类型       延迟(ms)     吞吐量(QPS)     连接数限制
─────────────────────────────────────────────
HTTP/1.1     50-100     5,000-10,000     每请求新建
HTTP/2       20-50      20,000-50,000    多路复用
gRPC         10-30      50,000-100,000   长连接池
WebSocket    5-15       100,000+         单连接持久
```

### 数据库：MySQL 8 + Redis 7 + MongoDB
```
MySQL
--  ACID事务保证（资金转账、消息已读状态）
--  复杂查询支持（用户搜索、群组成员）
--  成熟稳定，运维经验丰富
--  不适合：海量消息存储、高并发读写

Redis
--  极速读写（内存操作，<1ms响应）
--  数据结构丰富（String/Hash/List/Set/ZSet）
--  支持过期自动清理（Token有效期）
--  发布订阅功能（频道消息通知）
--  不适合：大规模持久化、复杂查询
应用场景：
--  JWT Token 黑名单（防重放攻击）
--  用户在线状态
--  消息未读计数

MongoDB
-- 灵活Schema（消息格式多变：文本/图片/语音/视频）
-- 水平扩展容易（分片集群）
-- 写入性能优秀（追加写入模式）
-- 适合海量数据（TB级消息历史）
-- 不适合：强事务需求、复杂JOIN查询

特性     MySQL           Redis         MongoDB 
数据模型  关系型            KV键值        文档型    
查询能力  强               弱             中        
写入性能  弱               强             强
扩展方式  垂直/分库分表      集群/哨兵       分片集群 
事务支持  ACID             单命令原子      多文档事务 
适用场景  用户/群组/关系     缓存/会话/计数   消息/日志/文件元数据

Kafka
核心优势：
-- 超高吞吐量：百万级 TPS
-- 顺序写磁盘 + 零拷贝技术
-- 适合 IM 消息洪峰场景

消息持久化与回溯：
-- 消息保留可配置（7天/30天/永久）
-- 支持重新消费历史消息
-- IM离线消息必选特性

分区并行消费：
-- 多消费者组水平扩展
-- 消息有序性保证（同一分区）

生态系统成熟：
-- Kafka Connect（数据管道）
-- KStreams（流处理）
-- 监控运维工具完善

Etcd
-- Go语言原生集成（go-zero 内置支持）
-- Raft一致性算法，CP模型，强一致性
-- 轻量级（资源占用小）
-- gRPC接口，性能优异
-- Kubernetes底层组件，云原生标配
```
